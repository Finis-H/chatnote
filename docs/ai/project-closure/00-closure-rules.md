# 00｜归档关闭执行规则

## 1. 基本原则

1. 本次任务的目标是归档、收束和建立重构基线，不是继续开发功能。
2. 任何“已实现”“已验证”“可运行”的结论都必须有可核验依据。
3. 不确定的内容必须写为“待确认”，并说明缺少什么证据；不得用合理推测填充事实。
4. 最终仓库应更清晰、更安全、更可回溯，而不是单纯文件更少。
5. 人负责产品边界、删除决策、对外状态和外部资源操作；Codex 负责审查、草拟、执行已批准的仓库内变更和验证。

## 2. 证据优先级

从高到低使用下列证据：

1. 在干净环境中的实际运行、构建、测试或 Trace 输出；
2. 当前分支的代码、配置、依赖锁文件和 CI 配置；
3. 已版本化的产品、架构、设计、评测文档；
4. 可定位的 Figma 文件、Release、截图、视频或测试夹具；
5. 旧讨论、历史说明和执行人的推断。

低优先级证据不得覆盖高优先级证据。若文档与代码不一致，应记录差异，不得偷偷修改历史结论。

## 3. 严禁动作

在任何工作包中，除非该工作包明确允许且用户已批准，不得：

- 删除、移动、覆盖仓库文件；
- 执行批量清理、强制重置、重写 Git 历史或清理未跟踪文件；
- 运行会修改生产数据、用户数据或本地持久化记忆的脚本；
- 上传、公开或复制包含密钥、个人数据、真实对话、日志或数据库内容的资产；
- 关闭部署、域名、云资源、CI、第三方账号或撤销密钥；
- 将未实现的 Figma 原型、规划能力或测试设想描述为已交付。

禁止使用的高风险命令包括但不限于：git reset --hard、git clean、强制推送、无清单的递归删除。

## 4. 删除与归档门槛

某候选项只有同时满足以下条件，才能进入“可删除”状态：

1. 已在代码、配置、构建脚本、CI 和文档中检索其引用；
2. 不承担动态加载、迁移、兼容、许可证、发布或作品集证据职责；
3. 其删除理由、影响范围和验证方式已写入 02-audit-manifest.md；
4. 用户已明确批准该项或所在批次；
5. 删除后已完成约定的构建、测试或关键流程验证。

若任何条件不能确认，状态必须为“待人工确认”或“保留”，不得删除。

## 5. Git 与版本控制

- 归档前后均应使用可读、可追溯的 Git 标签；
- 不修改历史，不压缩历史，不因“整洁”删除历史提交；
- 每个清理批次应形成独立、可审查的提交；
- 不处理与归档任务无关的现有改动；
- 若工作区已脏，先记录差异并请用户决定纳入快照、提交、暂存或排除。

## 6. 安全与隐私

- 只记录密钥位置、处理状态和服务名称，绝不把真实密钥写入归档文档；
- 对截图、日志、Trace、夹具和数据库导出进行个人数据与敏感内容检查；
- 未经批准，不上传任何本地数据或运行产物；
- 外部资源的关闭、删除、撤销、费用变更必须逐项得到用户确认。

## 7. 阶段输出要求

每个工作包完成后必须：

1. 生成或更新该工作包指定的 docs/archive/ 文件；
2. 更新 PROJECT_CLOSURE_STATE.md；
3. 报告执行范围、证据、验证结果、未决项和下一步确认门；
4. 停在当前阶段，不主动执行下一个工作包。

## 8. 文档语言与命名

- 面向用户和作品集的内容使用简体中文；
- 必要技术术语保留英文，例如 Agent、Trace、WebSocket、API、CI、Build、Release、Eval；
- 文件名采用既有英文命名，以便命令和链接稳定；正文使用简体中文；
- 代码路径、命令、提交号、版本号和原始错误信息保持原样。
