# 04｜执行已批准的清理

## 1. 目标

仅依据已经获得用户批准的清理清单，分批处理明确无价值或应移出主仓库的内容，并通过验证证明清理未破坏最终归档版本。

## 2. 前置条件

- 03-audit-only 已完成；
- 02-audit-manifest.md 中每个待处理项均有明确的“用户审批”结果；
- PROJECT_CLOSURE_STATE.md 已记录本次允许执行的清理批次；
- 用户已授权执行该批次。

## 3. 允许动作

- 处理已批准的低/中/高风险清理项；
- 删除或移动已批准的文件；
- 为仍需要保留的大文件创建归档索引或移动到用户指定位置；
- 更新受影响的引用、文档目录或忽略规则，但不得借机重构业务代码；
- 执行约定的 Build、Test、静态检查或关键流程验证；
- 创建 docs/archive/04-cleanup-execution-log.md 并更新状态文件。

## 4. 禁止动作

- 处理未批准、待确认或超出本批次范围的项目；
- 批量删除未跟踪文件或使用模糊匹配删除；
- 修改 Git 历史、强制推送或重置工作区；
- 清理未在审查清单中登记的代码；
- 把“清理失败”通过补写业务逻辑或大规模重构掩盖。

## 5. 分批执行流程

对每一个批次重复以下步骤：

1. 复核本批次的审批 ID、范围、风险和验证方式。
2. 记录操作前的 Git 状态与涉及文件清单。
3. 只处理批准项。
4. 立即执行约定验证；若验证失败，停止本批次，不继续处理后续项。
5. 记录变更、验证结果、失败情况、回退建议和剩余项。
6. 更新 02-audit-manifest.md、04-cleanup-execution-log.md 和 PROJECT_CLOSURE_STATE.md。
7. 向用户报告，等待批准后再执行下一批次。

## 6. 特殊规则

### 6.1 安装包与构建产物

- 若 Build 可复现且产物无独特展示价值，可删除仓库内副本；
- 若它是最终可演示版本，应先确认 Release 或受控存储位置、版本和校验值；
- 不能仅因文件体积大而删除唯一交付证据；
- 不要将大二进制直接迁入 docs/archive/。

### 6.2 文档与截图

- 删除旧文档前，先检查是否被 README、作品集、评测报告或外部链接引用；
- 若文档是历史决策证据，应保留或明确标记“历史版本”，而非直接抹除；
- 截图、Trace 和测试记录必须先完成敏感信息检查。

### 6.3 代码、依赖与配置

- 代码模块删除后必须至少完成静态检索、Build 和相关测试/关键路径验证；
- 依赖删除属于高风险变更，除非用户明确批准，否则只记录，不在归档阶段处理；
- 配置、迁移、插件注册、动态加载和权限相关内容默认保守处理。

| 批次 | 执行日期 | 审批 ID | 实际处理项 | 变更摘要 | 验证 | 结果 | 后续/回退建议 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| A | 待填写 | CLN-001 | 待填写 | 待填写 | 待填写 | 通过/失败/部分通过 | 待填写 |

每个失败或部分通过项必须额外写明：

- 失败的命令或验证路径；
- 是否已停止后续清理；
- 是否需要回退；
- 需要用户决定的选项。

## 8. 退出条件

所有已批准批次均有执行记录和验证结果；未批准或验证失败项仍保持原状并在状态文件中可见。
