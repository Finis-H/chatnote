# 01｜定义关闭边界与冻结项目

## 1. 目标

将 Vault OS 从“持续开发中的项目”切换为“归档关闭中的产品资产”，明确停止什么、保留什么、未来如何重启。

本工作包只建立边界与记录现状，不清理文件、不修改业务代码、不关闭任何资源。

## 2. 允许动作

- 只读检查 Git 状态、分支、提交、目录和已有文档；
- 创建或更新 docs/archive/00-closure-scope.md；
- 创建或更新 docs/archive/PROJECT_CLOSURE_STATE.md；
- 列出需要用户确认的项目状态、对外展示与资产保留决策。

## 3. 禁止动作

- 删除、移动、归档任何代码或文件；
- 建立 Git 标签、提交、分支或 Release；
- 修改 README 对外表述；
- 关闭外部服务或撤销密钥。

## 4. 执行步骤

1. 检查仓库根目录、当前分支、工作区状态、最近提交和已有文档。
2. 识别当前项目是否存在未提交改动、未跟踪文件、构建产物、外部部署线索和敏感配置线索。
3. 阅读已有 README、PRODUCT、ARCHITECTURE、Eval、Figma/设计索引等文档；若文件不存在，记录“未发现”，不要补写为事实。
4. 在 docs/archive/00-closure-scope.md 中填写关闭范围模板。
5. 初始化或更新 PROJECT_CLOSURE_STATE.md，将当前工作包标记为“待人工确认”。

## 5. 归档产物模板：00-closure-scope.md

该文件必须包含以下章节：

### 5.1 关闭声明

- 关闭类型：归档式关闭 / 暂停迭代 / 为重构保留基线；
- 生效日期；
- 当前负责人；
- 关闭原因：只记录已确认的原因；
- 非目标：例如“不在本阶段新增功能、重做 UI 或修复全部技术债”。

### 5.2 产品资产保留目的

分别说明 Vault OS 在以下用途中的保留方式：

| 用途 | 是否保留 | 保留内容 | 访问方式 | 备注 |
| --- | --- | --- | --- | --- |
| 作品集展示 | 待确认 | 待确认 | 待确认 | 不能展示敏感数据 |
| 产品重构 | 待确认 | 待确认 | 待确认 | 需保留真实现状 |
| 技术复盘 | 待确认 | 待确认 | 待确认 | 保留决策与证据 |

### 5.3 范围边界

| 类别 | 本阶段处理 | 不处理 | 决策/原因 |
| --- | --- | --- | --- |
| 代码与文件清理 | 审查后分批处理 | 未确认模块 | 待填写 |
| 产品文档 | 建立事实基线 | 重写未来 PRD | 待填写 |
| Figma/设计 | 建立链接与差异记录 | 为了归档重画 UI | 待填写 |
| 部署与密钥 | 清单与确认 | 未批准的关闭动作 | 待填写 |
| 新功能与缺陷 | 仅记录 | 不在旧架构修复 | 待填写 |

### 5.4 未来重构触发条件

至少写明：

- 新的目标用户和 MVP 范围已经冻结；
- 是否保留现有代码的决定已经完成；
- 架构与权限/记忆/插件边界已重新评审；
- 可投入的时间、预算或协作者条件；
- 重构仓库、分支或项目的启动方式。

### 5.5 必须由用户确认的决策

至少列出：

1. 仓库是否继续公开，以及是否在 README 中标记 Archived；
2. 最终可演示版本、截图、Figma 链接与安装包的保留位置；
3. 是否存在需要停止付费或自动运行的外部服务；
4. 清理前标签和最终标签的命名；
5. 哪些内容可用于 AI PM 作品集。

## 6. 退出条件

只有当用户确认关闭声明、保留用途、对外状态和标签命名策略后，才能进入 02-final-snapshot。
