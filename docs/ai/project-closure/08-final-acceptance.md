# 08｜最终验收与归档完成

## 1. 目标

确认 Vault OS 已完成归档式关闭：最终状态可回溯、清理有记录、文档反映事实、外部风险已处理或明确交接，并建立最终归档标签。

## 2. 前置条件

以下工作包应已完成或有明确“不适用/待用户线下确认”记录：

- 01-scope-and-freeze；
- 02-final-snapshot；
- 03-audit-only；
- 04-approved-cleanup；
- 05-current-state-baseline；
- 06-decommission；
- 07-retrospective-and-sop。

## 3. 允许动作

- 执行最终验证；
- 检查文档与链接；
- 创建 docs/archive/08-closure-report.md；
- 在用户确认后建立最终归档标签；
- 在关闭范围已批准的前提下更新 README 的归档说明；
- 更新 PROJECT_CLOSURE_STATE.md。

## 4. 禁止动作

- 为了通过验收而临时实现新功能、伪造截图、补造 Eval 结果或隐瞒失败；
- 处理未批准的清理项；
- 未经确认创建公开 Release、改变仓库可见性、关闭外部资源或推送远端；
- 删除任何最终证据。

## 5. 最终验收清单

### 5.1 版本与可回溯性

- [ ] 已记录清理前标签、提交号和最终标签、提交号；
- [ ] Git 工作区状态已记录；
- [ ] 清理批次均有审批、执行与验证记录；
- [ ] 没有通过重写 Git 历史掩盖遗留内容。

### 5.2 可运行性与验证

- [ ] 按 01-final-snapshot.md 的说明，在干净环境或等效环境完成 Build/Test/启动验证；
- [ ] 若不能完整运行，已记录失败原因、影响范围、复现条件和后续建议；
- [ ] 核心演示路径的截图/视频与实际版本一致；
- [ ] 验证没有使用真实密钥、个人数据或未经授权的外部服务。

### 5.3 文档与事实一致性

- [ ] README、产品说明、架构现状、功能状态和技术债文档不存在明显冲突；
- [ ] 已实现、部分实现、仅设计、已放弃的能力被正确区分；
- [ ] Figma 与实际实现的差异已记录；
- [ ] Eval 证据缺口已诚实标注；
- [ ] 归档文档链接与资产索引有效。

### 5.4 安全与外部资源

- [ ] 密钥、Token、配置、日志和截图已检查；
- [ ] 外部部署、域名、CI、数据库、存储和费用项目已处理或有用户确认的保留决定；
- [ ] 任何未能在线处理的事项已进入未决项，并给出负责人和下一步。

### 5.5 重构准备度

- [ ] 06-tech-debt-and-rebuild-backlog.md 已区分保留、重写、延后和放弃；
- [ ] VAULT_OS_RETROSPECTIVE.md 已沉淀关键经验；
- [ ] PRODUCT_REBUILD_SOP.md 已能独立用于新项目；
- [ ] 关闭范围已写明未来重构的触发条件。

## 7. 最终标签与 README

只有用户确认验收结论后，才可建立最终归档标签。标签建议使用稳定命名，例如：

- vault-os-archive-final
- vault-os-archive-YYYYMMDD

若用户选择继续公开仓库，README 顶部可采用以下简短表述，并按真实链接替换占位符：

~~~text
> 项目状态：已归档（Archive）。
>
> Vault OS 不再在当前架构上持续迭代。仓库保留最终可验证版本、产品现状与重构基线；
> 计划与实现差异请参阅 docs/archive/05-design-implementation-gap.md，
> 重构入口请参阅 docs/archive/06-tech-debt-and-rebuild-backlog.md。
~~~

不得在 README 中宣称未验证能力已完成，也不得承诺具体重启时间。
