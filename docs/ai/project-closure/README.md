# Vault OS 归档式关闭工作包

## 1. 使用目的

本目录是一套由人主导、由 Codex 协助执行的“归档式关闭”工作包。它适用于 Vault OS 当前停止继续迭代、但仍需保留为：

- 可回溯的最终产品版本；
- AI 产品经理作品集中的真实案例；
- 后续产品重构的事实输入；
- 可复用的项目方法论来源。

本工作包不是一次性清仓指令。执行时必须按顺序完成一个阶段、更新状态文件、完成阶段验收，再进入下一阶段。

## 2. 与归档产物的目录关系

本目录存放“如何执行”的流程说明：

| 目录/文件 | 作用 |
| --- | --- |
| docs/ai/project-closure/ | Codex 执行规则、阶段任务、输入输出和验收条件 |
| docs/archive/ | 执行过程生成的真实清单、基线、报告和复盘 |
| 根目录 README.md | 对外入口；归档完成后应说明项目状态与可验证范围 |

不要把构建缓存、安装包、视频、重复截图或旧代码简单地移动到 docs/archive/。归档文档应保存“资产在哪里、为何保留、如何验证”，大文件应放入 Release 或受控存储位置。

## 3. 执行顺序

| 顺序 | 工作包 | 允许的主要动作 | 人工确认门 |
| --- | --- | --- | --- |
| 0 | 00-closure-rules | 阅读规则 | 无 |
| 1 | 01-scope-and-freeze | 只读审查、创建关闭范围文档 | 关闭边界确认 |
| 2 | 02-final-snapshot | 记录、导出、建立清理前快照 | 标签与最终资产确认 |
| 3 | 03-audit-only | 只读审查、生成清理候选清单 | 每项清理决定确认 |
| 4 | 04-approved-cleanup | 仅处理已批准项，分批验证 | 每一清理批次确认 |
| 5 | 05-current-state-baseline | 编写事实基线与重构 Backlog | 内容事实确认 |
| 6 | 06-decommission | 检查并关闭外部资源 | 每个外部动作确认 |
| 7 | 07-retrospective-and-sop | 完成复盘与通用 SOP | 无；不得虚构事实 |
| 8 | 08-final-acceptance | 最终验证、关闭报告、最终标签 | 最终归档确认 |

## 4. 每次交给 Codex 的标准指令

执行任一工作包时，使用以下指令，并将“XX”替换为对应文件名：

~~~text
请执行 Vault OS 归档式关闭工作包 XX。

请先阅读：
1. docs/ai/project-closure/README.md
2. docs/ai/project-closure/00-closure-rules.md
3. docs/ai/project-closure/XX
4. docs/archive/PROJECT_CLOSURE_STATE.md

严格遵守当前工作包的允许动作、禁止动作和人工确认门。
只依据仓库、运行结果、已有文档和可核验资产陈述事实；不确定内容必须标记“待确认”。

完成后：
1. 写入本阶段要求的归档产物；
2. 更新 docs/archive/PROJECT_CLOSURE_STATE.md；
3. 汇报执行内容、验证结果、风险和下一步需要我确认的事项；
4. 不得自行跨越人工确认门。
~~~

## 5. 状态管理规则

每次执行前后都更新 docs/archive/PROJECT_CLOSURE_STATE.md。该文件是关闭过程的唯一进度源，不依赖聊天记录。

状态字段至少包括：

- 当前工作包与阶段状态；
- 清理前标签与最终归档标签；
- 已批准的清理批次；
- 外部资源处理状态；
- 未决事项和负责人；
- 最近一次验证结果与日期。

## 6. Vault OS 已知背景（执行前必须核验）

以下内容来自既有项目讨论，只能作为审查线索，不能直接当作实现事实：

- 产品定位为本地长期记忆 AI Agent 工作台；
- 关注记忆、工具/插件执行、Trace 可观测性与权限治理；
- 已讨论 WebSocket、低置信度记忆自动采纳、敏感操作确认和插件权限等能力；
- 曾规划四张 Figma 关键帧：主控终端、Agent 执行、记忆/权限审核、插件中心/设置；
- Figma 设计与实际前端曾存在布局和能力差异。

执行人必须以代码、运行结果、测试记录和可访问的设计资产验证这些描述，并在现状基线中标注真实状态。

## 7. 完成定义

只有同时满足以下条件，Vault OS 才可标记为“归档关闭完成”：

1. 存在清理前快照与最终归档标签，且提交号已记录；
2. 每个删除、移出或保留决定均有记录和验证；
3. 有真实的功能状态、架构现状、设计差异和重构 Backlog；
4. 外部服务、密钥、个人数据和自动化任务已检查并处理；
5. 最终版本按文档可构建、可启动或明确说明其不可运行原因；
6. 已完成 Vault OS 复盘和可复用的重构 SOP；
7. docs/archive/08-closure-report.md 已通过最终验收。
