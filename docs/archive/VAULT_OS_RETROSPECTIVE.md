> 本文完成 `07-retrospective-and-sop` 的 Vault OS 专属复盘。时间边界为清理前标签 `vault-os-pre-archive-20260716` 对应的源码，以及其后的归档记录。文中将可回溯记录标为“事实”，基于这些记录的解释标为“判断”，未来动作标为“建议”。未运行新的 Build、测试、服务、Docker、Tauri 或安装包，也未读取 `vault/`、运行时数据库、日志、真实对话或 Trace。

## 1. 项目摘要与归档结论

### 事实

- Vault OS 的定位是本地优先的长期记忆 AI Agent 工作台，围绕记忆、RAG、Agent 与工具/插件执行、Trace 可观测性、权限治理和桌面工作台体验组织。依据见 `00-closure-scope.md`、`03-feature-status.md` 与 `04-architecture-as-is.md`。
- 2026-07-16 已确认停止在既有架构上继续增量迭代，转入归档、复盘和重构准备；关闭原因是既有架构上的继续迭代收益较低。见 `00-closure-scope.md` 的 OQ-CLOSE-001。
- 有记录支撑的验证范围仅包括：隔离环境中的前端生产 Build 通过，以及临时 `VAULT_ROOT` 下最小后端启动后存活 8 秒并受控停止。完整后端单进程测试为 38 通过、11 错误；画像导入独立测试未执行，Web 搜索独立测试为 5 通过、2 失败。见 `01-final-snapshot.md`。
- Windows/Tauri 交付仅保留为未验证的候选和重建线索；未完成 Tauri bundle、签名、安装、卸载或升级验证。当前 sidecar 与 `bin/_internal/` 仍保留，私有 NSIS 归档只记录了标识与哈希，不在仓库记录位置。见 `01-final-snapshot.md`、`REBUILD_ENVIRONMENT.md` 与 `04-cleanup-execution-log.md`。
- 公开候选资产限于经确认的源代码、产品/架构/案例文档、脱敏截图和四张 synthetic、redacted、fixture-only 的 Figma 静态导出。运行时数据、数据库、日志、真实对话、Trace、密钥、未审核安全材料和未验证交付物不属于公开展示边界。见 OQ-CLOSE-003、OQ-CLOSE-007 与 `01-final-snapshot.md`。

### 判断

Vault OS 已形成可回溯的本地 AI 产品原型和重构输入，但未形成可宣称为完整回归通过、已安装验证或生产运行的最终交付。归档的价值在于保留边界、证据和失败，而不是把原型包装为已完成产品。

### 建议

任何后续介绍应使用“归档的本地 AI 工作台原型/重构基线”这一口径，并把已验证范围、未验证范围和展示资产边界一并给出。

## 3. 关键主题复盘

### 3.1 产品范围与迭代节奏

**事实：** 当前归档基线同时覆盖记忆、RAG、Agent、工具/插件、Trace、权限、多个前端视图和 Windows 桌面候选；`03-feature-status.md` 将其中多数能力标为“部分实现”。

**判断：** 项目的产品叙事覆盖了一个完整本地 AI 工作台所需的多个高风险域，而可验证证据集中在前端 Build、最小启动、静态代码/截图和部分测试记录。范围与证据积累节奏没有保持同步。

**建议：** 后续重构只选择一个可演示闭环作为 MVP，例如“受控本地知识检索 + 明确来源 + 可观察失败”，再按同一证据标准扩展记忆或 Agent，而不是并行承诺多个治理能力。

### 3.2 记忆、Agent、工具/插件、Trace 与权限边界

| 主题 | 事实 | 判断 | 建议 |
| --- | --- | --- | --- |
| 长期记忆 | 当前实现含缓冲、待审、合并、过期与画像导入两阶段路径；Eval 已记录 PII、偏见与失败可见性问题。 | 审核 UI 是必要交互，不构成对长期写入风险的充分控制。 | 用数据等级、来源、置信度、拒绝理由、保留期和可撤销性定义写入契约。 |
| RAG | 当前有本地检索、来源返回和向量库路径；Eval 记录 RAG 注入、冲突和 PII 最小披露未通过。 | 检索文本尚不能被视为已隔离的不可信输入。 | 在生成前明确包装不可信内容，在 UI 中显示冲突与出处，并用夹具验证输出最小披露。 |
| Agent 与工具 | 当前有意图门控、JIT/DAG、线程池执行和工具注册/执行。 | 编排、取消、错误归因和最终答复之间缺少已验证的一致终态。 | 以任务 ID、超时、取消、部分成功和工具错误对象建立稳定状态机。 |
| 插件与权限 | 当前有动态发现、权限识别、脱敏预览、一次/会话级允许和不可信输出包装。 | 动态加载、依赖与卸载数据副作用不能靠静态检索证明安全。 | 以 manifest、信任级别、独立进程/沙箱、权限账本和可回滚卸载流程重建边界。 |
| Trace | 当前有数据库 Trace、树构造、API 和前端展示路径。 | 没有新的端到端复验，且失败/取消语义仍为重构项。 | 将 Trace 作为运行契约的证据层，保存可复放的脱敏夹具，而不是只作为展示组件。 |

### 3.3 设计资产与真实实现

**事实：** Figma 导出及其 QA 仅证明四张 synthetic、redacted、fixture-only 静态资产可用于作品集交接；当前仓库未发现可定位的 Figma 原文件 URL。实际前端有主控、记忆审核、插件管理、画像导入等视图，实际截图展示了部分 UI 和 Trace。

**判断：** Figma 中更完整的 Agent 时间线、治理面板、风险状态和 fixture 插件状态是重构设计输入，不能计入运行产品能力。实际 UI 截图也只能证明已有界面资产，不证明当前端到端路径成功。

**建议：** 每张关键设计稿应链接到页面、事件、后端契约、正常/失败/权限状态和验收记录；没有运行证据的节点必须可见标记为“设计中”或“fixture”。

### 3.4 Eval、测试失败与验证缺口

**事实：** 最近完整单进程后端测试未通过，原因包含跨模块 `sys.modules` stub 污染；画像导入独立运行在隔离环境因 `PermissionError` 未执行；Web 搜索独立测试有两项防编造文案断言失败。`docs/EVAL_REPORT.md` 与 `docs/ARCHITECTURE.md` 的 Eval 汇总数字也存在待人工确认冲突。

**判断：** 当前测试和 Eval 是高价值的风险证据，但不是可用于发布判断的统一质量门。保留失败记录比用汇总数字掩盖差异更有价值。

**建议：** 重构先修复测试隔离和固定夹具，再建立“功能断言 + 风险 Eval + 脱敏 Trace/截图”的证据包；在原始评测口径确认前，不合并两份 Eval 汇总数字。

### 3.5 Windows 交付、环境与清理

**事实：** `REBUILD_ENVIRONMENT.md` 保存了重建所需的解释器/工具版本、锁/清单哈希、脱敏依赖清单和侧车前提。C-1 已清理 `.venv/` 与 `chat-ui/src-tauri/target/`；两个 Windows staging 目录因文件名级预检命中运行时数据线索，未读取、未删除，已转入 OQ-DECOM-004。

**判断：** 清理前的环境快照有效地把“可再生缓存”和“可能含用户数据的运行时材料”区分开；但构建候选仍不能替代干净 Windows 的实际交付验证。

**建议：** 以后每次清理前先做脱敏环境快照、路径级审计与私有资产哈希核验；对任何命中配置、聊天、数据库、Trace 或日志线索的目录立即停止自动清理并转给人决定。

## 6. 重构前必须改变的做法

1. **先冻结可验收的 MVP。** 每个能力声明必须包含目标用户、场景、非目标、失败行为和验收证据；未冻结的能力不进入实现。
2. **把高风险边界写成状态机与契约。** 记忆、RAG、Agent、工具、插件、权限、Trace 和临时会话不能只靠页面或函数名表达语义。
3. **维护设计—实现—证据映射。** Figma、代码和验收材料必须三向链接；静态设计资产始终保留其非运行态标记。
4. **将测试失败当作产品风险输入。** 建立隔离测试、固定夹具和最小 Eval 集，再讨论功能扩张或桌面发布。
5. **把环境、快照与关闭当作产品工程的一部分。** 每次可交付候选都要有可重建环境、可定位基线、私有资产边界和退出/回退说明。
