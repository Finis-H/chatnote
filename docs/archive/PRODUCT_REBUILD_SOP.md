# 产品重构与新 AI 产品 SOP

> 本 SOP 面向未来的 Vault OS 重构，以及单人或小团队与 Codex 协作的新 AI 产品。它把 Vault OS 归档中已有的快照、审计、环境、状态基线机制抽象为可复用工作法；不授权删除数据、发布、推送、外部账号操作或绕过人工确认。

## 1. 使用范围与基本规则

### 1.1 适用对象

适用于同时包含模型调用、Agent、Tool、Memory、RAG、插件、权限或本地数据的产品。若产品只含普通前端页面，可删减高风险边界章节，但仍保留需求、实现与证据映射。

### 1.2 单一事实源

每个产品从立项开始维护下列相互链接的事实源。文档可以换名，但职责不能缺失。

| 事实源 | 必须回答的问题 | Vault OS 可复用机制 |
| --- | --- | --- |
| 机会与 MVP 基线 | 为谁解决什么问题；本期不做什么；成功如何判定。 | 关闭范围中的“关闭原因/非目标”结构可复用为立项边界。 |
| 需求与决策记录 | 每个需求、取舍、所有者和确认时间是什么。 | `PROJECT_CLOSURE_STATE.md` 的状态、未决项、负责人和时间线结构。 |
| 设计—实现映射 | 每个用户场景对应哪些页面、状态、事件、API/数据契约与代码。 | `05-design-implementation-gap.md` 的资产/证据边界和场景对照结构。 |
| 验收证据 | 哪条命令、夹具、Trace、截图或人工脚本支持哪项结论。 | `01-final-snapshot.md` 的验证范围、结果与限制格式。 |
| 环境与快照 | 如何重建环境、定位候选版本、恢复或审计保留资产。 | `REBUILD_ENVIRONMENT.md`、Git 快照标签与 `02-audit-manifest.md`。 |

### 1.3 人与 Codex 的职责边界

| 决策或动作 | 人必须确认 | Codex 可协助但不能自行决定 |
| --- | --- | --- |
| 产品目标、用户、MVP、P0/P1 取舍、成功标准 | 是 | 访谈材料整理、需求草案、矛盾和风险审查。 |
| 数据分类、记忆写入、保留期、删除、备份、隐私例外 | 是 | 数据流梳理、策略草案、测试与审计清单。 |
| Agent 可执行动作、工具能力、插件信任级别、权限粒度 | 是 | 威胁建模、契约草案、脱敏预览、回归测试。 |
| 对外发布、公开资产、安装包、域名、密钥、账单、外部资源 | 是 | 发布清单、静态核对、证据汇总；不得代为发布或变更。 |
| 小切片代码、测试、文档、可逆的仓库内变更 | 在已冻结范围内授权 | 实现、代码审查、测试设计、执行验证并报告限制。 |

## 2. 端到端工作流

每一阶段都必须留下输入、输出、退出条件和最小验收。阶段未满足退出条件时，不以“继续实现”绕过问题。

| 阶段 | 输入 | 人的确认 | Codex 可执行工作 | 最小产出 | 退出条件 | 最小验收标准 |
| --- | --- | --- | --- | --- | --- | --- |
| 0. 机会定义 | 用户问题、已有流程、约束、成本和竞品线索。 | 目标用户、要解决的问题、明确非目标。 | 整理机会假设、风险、替代方案和待验证问题。 | 一页产品机会说明。 | 问题、用户和“不做什么”能在一页中说清。 | 至少一个主场景、一个失败/不适用场景和一项可观察成功信号。 |
| 1. MVP 冻结 | 已确认机会、时间/预算、技术约束。 | P0/P1、MVP 边界、延期项、上线/停止标准。 | 拆分用户故事、建立验收矩阵、指出范围冲突。 | 版本化需求基线与变更规则。 | 每个 P0 都有用户价值、正常/异常路径和验收方式。 | 不存在“先做出来再定义成功”的 P0；变更必须进入决策记录。 |
| 2. 高风险方案 | 需求基线、数据流、平台范围、外部依赖。 | 数据边界、权限模型、架构取舍、目标平台与成本。 | 方案对比、威胁建模、状态机/接口草案、依赖审查。 | 架构决策记录和高风险契约。 | Memory、RAG、Agent、Tool、插件、Trace、权限的所有权与失败路径明确。 | 每个高风险域至少有一张状态/数据流图和一组可测试的不变量。 |
| 3. 设计—代码—验收映射 | 用户流、架构契约、组件库/设计稿。 | 主路径、异常路径、信息与风险文案。 | 设计审查、组件状态清单、页面—事件—证据矩阵草案。 | 设计—实现—证据映射表。 | 关键页面的 loading、running、success、failed、needs_permission、empty 等状态齐全。 | 设计稿、实现任务与验收案例三者均可相互定位；静态 fixture 明确标注。 |
| 4. 小切片开发 | 已冻结的一项 P0/P1 切片、契约、映射和夹具。 | 切片完成定义与扩大范围的取舍。 | 实现、测试、审查、文档更新和本地验证。 | 可运行或可验证的最小切片及其证据包。 | 不新增未批准能力；异常和权限路径有行为证据。 | 见第 4 节的切片 Definition of Done。 |
| 5. Eval 与发布判断 | 多个完成切片、测试结果、真实或合成夹具、已知风险。 | 是否发布、修复、降级、重构或暂停；对外口径。 | 设计 Eval、执行授权的测试/Build、汇总证据与未决风险。 | Eval 报告、发布判定包、可排序 Backlog。 | 成功/失败阈值、已接受风险与回退路径已签字确认。 | 不把静态设计、单次 Build 或局部测试外推为完整交付。 |
| 6. 环境重建与归档关闭 | 停止或暂停决定、最终候选、数据与外部资产清单。 | 保留/删除/公开/私有边界、标签、外部资源和最终验收。 | 生成环境快照、审计清单、状态基线、复盘和关闭报告。 | 环境重建说明、快照标签、审计清单、当前状态与关闭报告。 | 版本、资产、证据、未决外部事项和回退方式均可定位。 | 仅处理已批准项；未决数据/外部资源默认私有保留，不因关闭文档而被标记为已关闭。 |

## 3. 高风险 AI 边界：先定契约，再写功能

下面的项目是未来 Vault OS MVP 或任何带自主性 AI 产品的“先决设计项”。若未完成契约，不应把相应能力列为可验收交付。

| 域 | MVP 前必须冻结的内容 | 最小验收证据 |
| --- | --- | --- |
| Memory | 数据等级、允许/拒绝/待审状态、来源、置信度、保留期、撤销/更正、失败反馈。 | 人工审核和自动规则的正反例夹具；写入失败、拒绝和撤销后的可观察状态。 |
| RAG | 数据来源、信任标签、不可信内容包装、冲突处理、引用显示、输出级最小披露。 | 注入、冲突、过度披露和空结果的 Eval 用例与脱敏输出证据。 |
| Agent | 任务 ID、计划、工具授权点、超时、取消、重试、部分成功和迟到结果丢弃规则。 | 一次正常、一次工具失败、一次取消/超时和一次权限等待的 Trace/测试夹具。 |
| Tool | 输入 schema、输出 schema、幂等性、外部副作用、错误对象、重试与审计字段。 | schema 验证、失败模拟、不可达依赖和副作用确认测试。 |
| 插件 | manifest、来源/信任级别、进程边界、依赖隔离、权限账本、数据目录、卸载与回滚。 | 第一方与第三方插件夹具；拒绝、仅本次、会话允许和卸载失败的回归记录。 |
| 权限 | 谁确认、确认对象、用途、脱敏参数预览、有效范围、过期、服务端校验和审计日志。 | 前后端一致的 permission token 测试；敏感参数不泄露的截图或夹具。 |
| Trace | 事件 schema、父子关系、终态枚举、PII 脱敏、保存期限与查询权限。 | 端到端事件序列、失败/取消一致性测试和脱敏审查。 |
| 临时会话 | 与主记忆和持久状态的隔离、迟到结果处理、可用工具白名单。 | 证明“不读主记忆、不写持久状态、结束后丢弃迟到结果”的自动化或可复现测试。 |

## 4. 小切片 Definition of Done

一个切片只有同时满足以下条件，才可进入 Eval 或下一切片：

1. **边界清楚：** 关联需求 ID、非目标、数据/权限等级和受影响状态已经记录。
2. **契约一致：** 页面状态、事件/API、后端状态机和错误对象不存在未说明的差异。
3. **正常与失败可观察：** 至少覆盖成功、失败和需要用户介入三类结果；AI/工具流程还应覆盖超时或取消。
4. **证据可回放：** 测试、脱敏夹具、命令输出、Trace 或截图能定位到提交与环境；静态稿不能替代运行证据。
5. **不扩大事实：** 未验证能力标为未验证；测试失败、限制和外部依赖记录在 Backlog 或状态文件中。
6. **可回退：** 有明确关闭开关、数据回退方案或“不可逆且已由人确认”的记录。

## 5. Eval、测试与发布判定

### 5.1 证据分层

发布判断必须区分下列证据，低层证据不能覆盖高层缺口：

1. 静态代码、设计稿与文档：证明意图或实现线索。
2. 单元/集成测试：证明固定夹具下的契约。
3. 隔离 Build、最小启动和端到端脚本：证明指定环境与路径。
4. 风险 Eval：证明 RAG 注入、隐私、工具失败、权限和状态表达等 AI 特有边界。
5. 人工验收与真实使用：只在授权、脱敏和数据边界明确时记录。

### 5.2 发布判定包

每个候选至少包含：版本/提交、环境快照、变更清单、需求—测试—证据矩阵、通过与失败明细、已接受风险、回退/暂停方法和对外表述。Windows 或桌面交付还必须补充干净环境 Build、签名归属、安装、启动、升级与卸载证据。

以下情况应阻止发布或改为“内部候选”：高风险域缺少失败路径证据；测试因共享状态而不可重复；RAG/Memory/插件权限缺少安全 Eval；Windows 安装尚未在隔离环境验证；或对外表述依赖静态设计、未审查截图、未确认的汇总数字。

## 6. 环境、快照、审计与关闭的可复用机制

### 6.1 环境重建快照

在清理可再生缓存、迁移工具链或制作桌面候选前，生成类似 `REBUILD_ENVIRONMENT.md` 的脱敏快照，至少包含：

- 解释器、包管理器、Node、Rust/目标平台工具版本；
- 依赖清单和锁文件哈希；
- 可安全公开的依赖版本；
- 重建命令、平台前置条件和已知未验证范围；
- sidecar、运行时数据、私有资产和密钥的明确排除项。

### 6.2 快照标签与审计清单

- 在可能影响可回溯性的变更前，建立可读标签并记录提交号；标签命名和创建须由人确认。
- 用审计清单逐项记录路径、引用、数据风险、保留/删除理由、验证方法和审批状态。
- 对命中聊天、配置、数据库、Trace、日志或未知运行时内容的路径，不读取内容来“证明可删”；默认停止、私有保留并请求人决定。

### 6.3 当前状态基线

维护一个类似 `PROJECT_CLOSURE_STATE.md` 的状态文件，作为唯一进度源：当前阶段、已完成产物、版本锚点、未决事项、负责人、外部资源、最近验证、下一步门槛。不要把聊天记录当作项目状态来源。

### 6.4 归档关闭门槛

只有在以下内容齐备后，项目才能进入最终关闭验收：最终候选可定位；事实基线和复盘已写入；每个清理/保留决定有证据与审批；外部资源逐项标记为已处理或待人工确认；可构建/可运行范围和不可验证原因已明确；公开/私有资产边界经过确认。未决外部事项不是“已关闭”。

## 7. 暂停、重构与停止的触发条件

应暂停当前实现并回到需求/方案阶段的信号包括：MVP 边界连续变化；高风险数据或权限路径无法给出状态机；测试失败无法隔离；静态设计与真实契约差距持续扩大；或交付成本超过已确认资源。

重构开始前，人必须确认：目标用户与 MVP、数据处理和保留规则、插件/工具可执行边界、目标平台与发布策略、可投入时间/预算、旧代码和运行时数据的复用/隔离策略、以及可接受的 P0/P1 风险。Codex 可以为这些决策准备证据、草案和验证，但不能替代确认。

## 8. 最小执行节奏

对于单人或小团队，推荐以一至两周为一个“证据切片”而非功能堆叠周期：周初冻结一个切片与验收；中间维护设计—代码—证据映射；周末运行授权的验证、记录失败和决定“发布/修复/降级/停止”。每个周期结束都更新状态基线和 Backlog，使下一周期从事实而非记忆开始。
