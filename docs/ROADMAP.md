# Vault OS Roadmap

> 文档定位：本路线图用于展示 Vault OS 从本地 AI 外脑到可信 Agent 工作台，再到可分发产品的阶段性产品判断。本文描述计划与目标，不代表所有能力已经完成。

## 路线原则

- **Local-first**：优先保证数据、记忆、知识库和插件运行在用户本地可控环境中。
- **Trust before autonomy**：在扩大 Agent 能力前，先补齐 Eval、权限、Trace、失败复盘和可撤销边界。
- **Portfolio clarity**：作品集表达应清楚呈现问题定义、产品取舍、验证指标和未完成事项，不夸大生产化状态。
- **A2A 作为长期路线**：Agent-to-Agent / A2A 只作为远期探索方向，近期不承诺完成。

## V0.1 可用的本地 AI 外脑

### 目标

让 Vault OS 成为一个可本地运行、可持续使用的 AI 外脑：用户可以在桌面工作台中完成 Chat、知识检索、长期记忆、Trace 观察和基础插件调用，并能理解系统正在做什么。

### 核心功能

- **Chat 主流程**：支持本地桌面工作台内的连续对话、临时会话和基础上下文组织。
- **RAG 知识检索**：围绕本地 `vault/` 管理知识文件、向量库和检索结果，让回答可结合用户自己的材料。
- **Memory 长期记忆**：区分一次性请求与可沉淀记忆，保留谨慎写入、待确认和来源意识。
- **Trace 可观测**：记录 AI/tool/action 的关键步骤、状态、耗时和失败信息，避免只展示最终回答。
- **插件基础能力**：支持插件发现、执行入口、权限确认和 AgentDock 面板承载。
- **作品集基础材料**：补齐 README、Product Case、Architecture、Eval Report 和截图/TODO 占位的基础入口。

### 成功指标

- 新用户能在 10 分钟内按 Developer Guide 启动后端和前端，并完成一次 Chat。
- 至少能完成一条端到端演示链路：导入知识材料、触发 RAG、生成回答、查看 Trace。
- 记忆写入有明确确认或规则边界，不把明显一次性问题直接固化为长期事实。
- 插件调用能展示插件来源、权限含义、执行状态和失败提示。
- 文档能让初次接触者在 30 秒内理解产品定位、核心能力和当前完成度。

### 风险

- RAG 召回质量不稳定，可能导致回答看似引用本地材料但实际依据不足。
- Memory 写入过度会污染长期画像，写入过少又无法体现长期协作价值。
- Trace 信息如果过于工程化，PM/非开发读者难以理解。
- 插件权限边界如果表达不清，容易让用户误以为第三方插件输出完全可信。
- 本地环境依赖较多，安装失败会影响第一印象。

### 不做什么

- 不承诺云端同步、多端协作或生产级账号体系。
- 不把所有聊天内容自动写入长期记忆。
- 不把插件输出当作可信系统指令。
- 不追求通用大模型聊天产品的完整功能覆盖。
- 不在 V0.1 中承诺 A2A、复杂多 Agent 编排或跨设备协作。

## V0.2 可信的 Agent 工作台

### 目标

把 Vault OS 从“可用的本地 AI 外脑”推进为“可信的 Agent 工作台”：用户不仅能让 AI 执行任务，还能评估结果、理解权限、追踪历史、复盘失败，并在高风险操作前保持控制权。

### 核心功能

- **Eval 体系**：建立面向 RAG、Memory、插件执行和任务完成质量的轻量评测集，覆盖正确性、引用可靠性、权限边界和失败解释。
- **权限中心**：集中展示插件权限、敏感操作、授权历史和撤销入口，让用户知道“谁在请求什么、为什么请求、影响范围是什么”。
- **任务历史**：沉淀 Agent 任务记录，包括输入、执行步骤、工具调用、权限确认、结果状态和相关 Trace。
- **失败复盘**：对失败任务提供原因分类、关键证据、可重试建议和用户可采取的下一步。
- **Trace 产品化**：从工程日志升级为可读的任务时间线，区分 `idle`、`running`、`success`、`failed`、`needs_permission` 等状态。
- **高风险确认统一**：删除、卸载、配置重载、记忆写入、权限授权等流程使用一致的确认与风险说明。

### 成功指标

- 关键任务失败后，用户能从任务历史中定位失败阶段、失败原因和下一步建议。
- Eval Report 至少覆盖 RAG、Memory、插件权限和 Trace 可读性四类场景，并明确通过/未通过样例。
- 权限中心能展示已授权插件、敏感权限、最近授权记录和撤销/调整路径。
- 高风险操作文案能说明操作对象、影响范围、是否可撤销和风险提示。
- 作品集中能清楚展示“为什么可信比更强自动化更优先”的产品取舍。

### 风险

- Eval 设计过重会拖慢产品迭代，过轻又无法支撑“可信”叙事。
- 权限中心如果只做展示、不支持有效撤销，会变成形式化安全。
- 任务历史涉及本地隐私数据，需要避免把敏感内容暴露到不必要的位置。
- 失败复盘如果解释过度，会让用户误以为系统总能准确诊断根因。
- Trace、任务历史和权限数据模型扩展可能影响现有后端 API 与前端状态管理。

### 不做什么

- 不把 Agent 默认升级为完全自主执行高风险任务。
- 不为了演示效果伪造 Eval 分数、用户数据或生产指标。
- 不引入复杂企业级权限模型，如多租户 RBAC、组织审计后台和云端策略下发。
- 不把失败复盘写成不可验证的“AI 自我解释”。
- 不把 A2A 作为 V0.2 的交付目标；最多保留接口边界和长期研究记录。

## V0.3 可分发产品

### 目标

让 Vault OS 从个人项目/作品集原型走向可被他人试用的本地优先产品：降低安装门槛，提供 Demo 数据和示例任务，形成插件分发与反馈闭环。

### 核心功能

- **安装包**：提供面向 Windows 的可安装版本或清晰打包产物，减少用户手动配置成本。
- **Demo 数据**：内置或可选导入演示知识库、示例记忆、Trace 示例和插件任务样例，便于快速体验核心价值。
- **插件市场**：提供本地优先的插件发现、安装、启用、卸载、权限说明和可信来源提示。
- **用户反馈**：建立轻量反馈入口，收集安装失败、RAG 质量、权限困惑、插件体验和任务失败样例。
- **首次体验流程**：围绕“导入材料 -> 提问 -> 查看依据 -> 授权插件 -> 复盘 Trace”设计一条可演示的新手路径。
- **作品集展示包**：整理 Demo 脚本、截图、产品说明、Eval 结果和关键取舍，形成可复用讲述材料。

### 成功指标

- 非项目作者能基于安装包或明确步骤完成首次启动，并跑通 Demo 任务。
- Demo 数据能够稳定展示 Chat、RAG、Memory、Trace 和插件权限的完整价值链路。
- 插件市场中的每个插件都有来源、权限、用途、风险说明和卸载路径。
- 用户反馈能沉淀为可分类的问题列表，并反向驱动 Roadmap 或 Eval 用例更新。
- 作品集材料能支持 5-8 分钟产品讲述，覆盖问题、方案、验证、风险和后续规划。

### 风险

- 打包链路可能暴露跨平台兼容、依赖体积和本地模型配置问题。
- Demo 数据如果过度设计，容易显得像脚本演示而不是真实产品能力。
- 插件市场会放大安全边界问题，需要避免第三方插件获得过高信任。
- 用户反馈入口如果收集敏感本地数据，可能违背 local-first 产品承诺。
- 分发产品会提高稳定性预期，现有原型级边界需要被清楚标注。

### 不做什么

- 不承诺生产级 SaaS、云同步、团队协作后台或付费商业化闭环。
- 不把 Demo 数据包装成真实用户案例或真实业务指标。
- 不开放未经权限审查的插件自动安装与自动执行。
- 不默认上传用户本地知识、记忆、Trace 或插件执行记录。
- 不把 A2A 写入 V0.3 必交范围；A2A 仍作为长期方向，在插件协议、任务边界和 Trace 模型成熟后再评估。

## 长期观察方向

- **A2A / Agent-to-Agent**：在单 Agent 可观测、权限边界和任务复盘足够稳定后，再评估多 Agent 协作、Agent 之间的权限委托和任务交接。
- **更成熟的个人知识操作系统**：围绕长期记忆、知识资产、任务历史和用户反馈建立更清晰的信息生命周期。
- **跨模型与本地模型体验**：在不写死具体模型的前提下，优化不同 LLM、embedding 和本地推理配置的兼容体验。
- **可验证 AI 产品质量体系**：把 Eval、Trace、用户反馈和失败复盘连接成持续改进闭环。
