# Vault OS Eval Report

> 文档定位：这是 Vault OS 的最小 AI 产品质量评测基线，用于评估本地优先 AI 工作台在知识问答、长期记忆、工具调用、插件安全和临时会话隔离上的可用性、可信度与风险边界。

## 评测原则

- 本报告只定义评测设计与判定标准，不记录未执行的结论。
- 未执行用例的 `实际结果` 列保持为空，不虚构通过率、失败率或质量结论。
- 评测关注 AI PM 视角的用户价值、信任边界、风险暴露和可恢复性，而不只是接口是否返回成功。
- 每条用例按风险选择适用证据进行人工复核，例如 Trace、权限提示、记忆记录、RAG 来源或 UI 状态；不要求每条同时具备所有证据。
- 备注中的 `当前可验证` 表示可在当前产品中人工执行并记录结果；`目标态/预期失败` 表示实现尚不完整，执行后如实记录风险或未通过，且不纳入通过数。
- 所有密钥、PII、插件和知识文本均使用虚构夹具；破坏性测试仅在独立 `VAULT_ROOT` 与一次性插件目录中执行。
- `实际结果` 推荐格式：`通过/未通过/阻塞；证据类型；Trace ID 或截图编号；一句结论`。
- RAG 仅使用当前可摄入的插件知识载荷或已存在的本地知识，不假定 README、Architecture 等仓库文档会自动进入索引。

## 证据留存口径

本报告包含第一轮人工评测结果与后续回归测试设计。由于部分用例来自历史手工测试，未完整保留 Trace、截图、夹具编号或数据库记录 ID，因此本报告不为这些历史结果补造证据编号。

证据状态分为：

- 已留存：存在 Trace、截图、日志、数据库记录或夹具文件，可复核。
- 历史人工执行，证据未留存：测试曾人工执行并记录结论，但原始 Trace/截图/记录 ID 未保留。
- 当前可复核：当前产品中可以按用例复跑并补充证据。
- 目标态/预期失败：能力尚未完整实现，记录为风险或后续改进项，不作为已验证通过能力宣传。
- 未执行：仅保留用例设计，不计入通过率或失败率。

`Evidence ID` 如有使用，仅表示本报告内的证据缺口追踪编号，不等同于系统 Trace ID、截图编号或数据库记录 ID。

## 风险等级

| 等级 | 含义 |
|---|---|
| P0 | 可能造成数据泄露、越权执行、误删共享数据或破坏核心安全边界。 |
| P1 | 可能造成长期记忆污染、错误决策、用户信任明显受损或关键流程不可恢复。 |
| P2 | 影响主要体验、可观察性或结果质量，但通常可由用户发现并修正。 |
| P3 | 轻微体验、文案、排序或可用性问题，不直接破坏核心边界。 |
